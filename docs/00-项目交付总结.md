# AssessV2 项目交付总结（重构后）

- 交付日期：2026-03-18
- 文档版本：v3.0
- 口径：以当前代码与迁移结果为准

## 1. 交付结论

本次交付完成了考核系统核心架构重建，系统从“年度中心”切换为“场次中心”，并完成前后端统一。

已落地关键能力：

- Root/Admin 按组织范围管理考核场次。
- 场次拥有独立周期与对象分组配置。
- 场次默认对象自动从组织架构生成，可增删，实体来源保持一致。
- 规则管理改为结构化编辑（分数计算两级 + 等第划分）。
- 规则绑定改为“选择即拷贝”，组织间互不干扰。
- 规则权限细化：Root 与 Admin 各自边界清晰。

## 2. 当前可用模块

后端模块（路由组）：

- `/api/auth`：登录、登出、改密
- `/api/org`：组织、部门、员工、职级等
- `/api/assessment`：场次、周期、对象分组、对象管理
- `/api/rules`：规则文件库、隐藏/恢复、绑定
- `/api/backup`：备份、下载、恢复
- `/api/system`：用户、角色组、设置、审计

前端模块（主导航）：

- 系统概览
- 考核管理
- 组织架构
- 规则管理
- 用户管理（Root）
- 备份恢复
- 审计日志
- 系统设置（Root）

## 3. 核心重构成果

### 3.1 业务模型重构

- 新增并启用场次模型：`assessment_sessions`。
- 每个场次独立管理：
  - 周期：`assessment_session_periods`
  - 对象分组：`assessment_object_groups`
  - 具体对象：`assessment_session_objects`
- 规则改造：
  - 文件库：`rule_files`
  - 组织级隐藏：`rule_file_hides`
  - 周期/对象分组绑定：`assessment_rule_bindings`

### 3.2 数据隔离与目录结构

- 场次数据目录：`data/{assessment_name}/`
- 场次规则目录：`data/rules/{assessment_name}/`
- 绑定规则时自动写入拷贝文件并落库

### 3.3 默认生成逻辑

创建场次时默认生成：

- 周期：`Q1/Q2/Q3/Q4/YEAR_END`
- 对象分组：
  - 团体：`dept`、`child_org`
  - 个人：`dept_main`、`dept_deputy`、`general_staff`、`child_leadership_main`、`child_leadership_deputy`
- 对象清单：
  - 本组织全部部门及人员
  - 下一级组织整体及其领导班子成员

### 3.4 规则管理结构化界面

- 分数计算采用两级结构：一级模块 + 二级子模块。
- 内置“额外加减分模块”，固定存在、不可删除、不占权重。
- 总分公式：`Σ(单模块分数 * 权重 / Σ总权重) + 额外加减分`。
- 计分方式支持：`direct_input`、`vote`、`custom_script`。

## 4. 权限模型交付结果

角色标准化：

- `root`
- `assessment_admin`
- `leader`
- `staff`

核心边界：

- Root：全组织可见可管；可新建基础规则；可删除非拷贝规则。
- Admin：仅本组织场次范围；仅可编辑/删除本组织拷贝规则；可隐藏非拷贝规则。

## 5. 与旧架构的关系

- 本次重构按“无需兼容旧结构”执行。
- 业务迁移 `0002_assessment_session_rebuild.up.sql` 已显式删除旧年度相关核心表。
- 旧“年度唯一场次”约束未引入当前模型（即不限制“同年同组织唯一场次”）。

## 6. 残留风险与后续建议

- 仓库仍有部分历史服务/模型文件，建议继续做代码物理清理，减少维护噪音。
- `/api/org/assessment-categories` 路由保留但不在新主流程使用，建议按是否继续保留统一处理。
- 建议补全场次与规则主流程的端到端自动化测试（创建场次 -> 绑定规则 -> 编辑拷贝规则）。

## 7. 验收要点（建议）

- Root/Admin 在同一场次下对规则文件操作权限差异可复现。
- 同一源规则被不同组织绑定后，编辑互不影响。
- 场次对象“重置默认”可回到组织架构快照集合。
- 顶部上下文切换后，规则绑定上下文同步生效。
