# 数据库设计更新说明

**更新日期**: 2026-03-11  
**版本**: 2.2

---

## 更新概述

根据用户需求，数据库设计已更新以支持：
1. **单个用户拥有多个角色**
2. **单个用户负责多个组织**
3. **自定义规则模块（custom）与跨周期/组织上下文计算**
4. **投票改为优良中差档位并通过规则映射得分**
5. **权重外加减分不占用权重**
6. **最高权限账号统一为 root**

---

## v2.2 主要补充（2026-03-11）

1. **score_modules 扩展**
- 增加 `module_code=custom`

- 最高权限账号统一为：`username=root`、`role=root`
- 默认密码：`#2026@hdwl`（首次登录强制改密）

## 主要变更

### 1. users 表结构调整

**变更前**：
```sql
CREATE TABLE users (
    ...
    role_id INTEGER NOT NULL,           -- 单一角色
    organization_id INTEGER,            -- 单一组织
    department_id INTEGER,              -- 单一部门
    ...
);
```

**变更后**：
```sql
CREATE TABLE users (
    ...
    -- 移除了 role_id, organization_id, department_id 字段
    -- 通过关联表实现多对多关系
    ...
);
```

### 2. 新增 user_organizations 表

支持用户与多个组织的关联关系。

```sql
CREATE TABLE user_organizations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    organization_type VARCHAR(20) NOT NULL,  -- group, company, department
    organization_id INTEGER NOT NULL,
    role_in_org VARCHAR(50),                 -- leader, admin, member
    is_primary BOOLEAN NOT NULL DEFAULT 0,
    created_by INTEGER,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE(user_id, organization_type, organization_id)
);
```

**功能特性**：
- 支持用户关联多个组织（集团/企业/部门）
- 每个关联可以指定在该组织中的角色（负责人/管理员/成员）
- 可以标记主要负责组织

### 3. user_roles 表增强

**变更前**：
```sql
CREATE TABLE user_roles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    role_id INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    ...
);
```

**变更后**：
```sql
CREATE TABLE user_roles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    role_id INTEGER NOT NULL,
    is_primary BOOLEAN NOT NULL DEFAULT 0,  -- 新增：标记主要角色
    created_by INTEGER,                      -- 新增：创建人
    created_at INTEGER NOT NULL,
    ...
);
```

**功能特性**：
- 支持用户拥有多个系统角色
- 可以标记主要角色（用于默认权限判断）
- 权限取所有角色的并集

## 使用场景示例

### 场景 1：用户同时负责多个企业

```sql
-- 用户张三同时负责企业A和企业B
INSERT INTO user_organizations (user_id, organization_type, organization_id, role_in_org, is_primary) VALUES
(1, 'company', 10, 'leader', 1),  -- 企业A负责人（主要）
(1, 'company', 11, 'leader', 0);  -- 企业B负责人
```

### 场景 2：用户拥有多个角色

```sql
-- 用户李四既是集团管理员，也是企业管理员
INSERT INTO user_roles (user_id, role_id, is_primary) VALUES
(2, 2, 1),  -- 集团管理员（主要角色）
(2, 3, 0);  -- 企业管理员
```

### 场景 3：用户在不同组织中担任不同角色

```sql
-- 用户王五在集团是成员，在部门A是管理员，在部门B是负责人
INSERT INTO user_organizations (user_id, organization_type, organization_id, role_in_org) VALUES
(3, 'group', 1, 'member'),
(3, 'department', 20, 'admin'),
(3, 'department', 21, 'leader');
```

## 权限判断逻辑

### 1. 角色权限

用户的最终权限 = 所有角色权限的**并集**

```
用户角色：[集团管理员, 企业管理员]
集团管理员权限：[user:*, org:*, score:*]
企业管理员权限：[employee:*, score:input]
最终权限：[user:*, org:*, score:*, employee:*]
```

### 2. 数据范围权限

用户可以访问的数据范围 = 所有关联组织的数据

```
用户关联组织：[企业A, 企业B, 部门C]
可访问数据：
- 企业A的所有数据
- 企业B的所有数据
- 部门C的所有数据
```

### 3. 组织内角色权限

在特定组织中的操作权限由 `role_in_org` 决定：

| role_in_org | 权限说明 |
|-------------|---------|
| leader | 负责人，拥有该组织的完全管理权限 |
| admin | 管理员，可以管理数据但不能修改组织结构 |
| member | 成员，只能查看数据 |

## API 查询示例

### 查询用户的所有角色

```sql
SELECT u.id, u.username, r.role_code, r.role_name, ur.is_primary
FROM users u
JOIN user_roles ur ON u.id = ur.user_id
JOIN roles r ON ur.role_id = r.id
WHERE u.id = ?;
```

### 查询用户负责的所有组织

```sql
SELECT 
    uo.organization_type,
    uo.organization_id,
    uo.role_in_org,
    uo.is_primary,
    CASE 
        WHEN uo.organization_type = 'group' THEN o.org_name
        WHEN uo.organization_type = 'company' THEN o.org_name
        WHEN uo.organization_type = 'department' THEN d.dept_name
    END AS org_name
FROM user_organizations uo
LEFT JOIN organizations o ON uo.organization_type IN ('group', 'company') AND uo.organization_id = o.id
LEFT JOIN departments d ON uo.organization_type = 'department' AND uo.organization_id = d.id
WHERE uo.user_id = ?;
```

### 查询用户在特定组织中的角色

```sql
SELECT role_in_org
FROM user_organizations
WHERE user_id = ? 
  AND organization_type = ? 
  AND organization_id = ?;
```

## 索引优化

新增的索引确保查询性能：

```sql
-- user_organizations 表索引
CREATE INDEX idx_user_organizations_user_id ON user_organizations(user_id);
CREATE INDEX idx_user_organizations_org_type ON user_organizations(organization_type);
CREATE INDEX idx_user_organizations_org_id ON user_organizations(organization_id);
CREATE INDEX idx_user_organizations_role ON user_organizations(role_in_org);
CREATE INDEX idx_user_organizations_is_primary ON user_organizations(is_primary);
CREATE INDEX idx_user_organizations_user_org ON user_organizations(user_id, organization_type, organization_id);

-- user_roles 表索引
CREATE INDEX idx_user_roles_is_primary ON user_roles(is_primary);
```

## 注意事项

1. **主要角色/组织标记**：
   - 每个用户应该有且只有一个主要角色（is_primary=1）
   - 每个用户应该有且只有一个主要负责组织（is_primary=1）
   - 应用层需要确保这个约束

2. **权限检查**：
   - 在应用层实现权限检查时，需要查询用户的所有角色
   - 数据范围过滤需要考虑用户的所有关联组织

3. **性能考虑**：
   - 用户角色和组织关系应该缓存在会话中
   - 避免每次请求都查询数据库

4. **数据一致性**：
   - 删除组织时，需要同步删除 user_organizations 中的关联记录
   - 删除角色时，需要同步删除 user_roles 中的关联记录

- ✅ 用户可以拥有多个系统角色

这种设计更符合实际业务场景，特别适合集团企业中一人身兼多职的情况。
