# 多级考核对象划分与规则复用设计

## 1. 背景与问题

当前系统的规则维度以 `year + period + object_type + object_category` 为主。  
随着集团/权属企业多级考核场景增加，`object_category` 同时承载了多种语义：

- 对象类型（团体/个人）
- 对象实体（部门/领导班子/人员）
- 所属组织层级（集团/权属企业/次级企业）
- 人员身份（正职/副职/一般人员）
- 被谁考核（本级/下一级/下两级）

结果是分类数量持续膨胀，规则难复用，维护成本高。

## 2. 设计目标

- 目标一：将“对象定义”与“考核关系”解耦，避免分类字段过载。
- 目标二：实现多级组织下规则复用，减少重复配置。
- 目标三：让“总分规则 + 模块规则 + 等第规则”以统一规则包管理。
- 目标四：支持按组织、周期、对象分群进行继承与覆盖。
- 目标五：可在现有系统上渐进迁移，降低改造风险。

## 3. 新模型总览

核心映射关系：

`考核主体(owner) + 周期(period) + 对象分群(segment) -> 规则包(bundle)`

其中：

- `owner`：发起考核的组织（集团、权属企业等）
- `period`：考核周期（Q1/Q2/Q3/Q4/YEAR_END 或其他）
- `segment`：被考核对象分群（由对象画像和相对关系计算）
- `bundle`：完整规则包（总分、模块、等第、排名）

## 4. 对象划分方式（对象画像）

将现有“单字段分类”升级为“对象画像维度”。建议至少包含：

- `subject_type`：`team | individual`
- `entity_kind`：`department | leadership_team | person`
- `role_series`：`leadership | department | general`
- `role_rank`：`main | deputy | member`
- `belong_org_id`：对象归属组织
- `belong_org_level`：对象归属层级（0/1/2...）
- `relation_to_owner`：相对考核主体关系  
  `self | child_org | grandchild_org | descendant_org`

说明：

- `relation_to_owner` 是复用关键。  
  例如“集团考核下一级企业班子”和“权属企业考核下一级企业班子”是同一关系模板，只是 owner 不同。

## 5. 对象分群（Segment）定义

规则不再直接绑定 `object_category`，而是绑定标准分群。

示例分群：

- `SEG_SELF_DEPT_TEAM`：本级部门团体
- `SEG_SELF_DEPT_PERSON_MAIN`：本级部门个人正职
- `SEG_SELF_DEPT_PERSON_DEPUTY`：本级部门个人副职
- `SEG_SELF_DEPT_PERSON_MEMBER`：本级部门个人一般人员
- `SEG_CHILD_LEADERSHIP_TEAM`：下一级企业领导班子团体
- `SEG_CHILD_LEADERSHIP_PERSON_MAIN`：下一级企业班子成员正职
- `SEG_CHILD_LEADERSHIP_PERSON_DEPUTY`：下一级企业班子成员副职
- `SEG_CHILD_LEADERSHIP_PERSON_MEMBER`：下一级企业班子成员一般人员

每个分群由条件表达式定义，例如：

`subject_type=individual AND relation_to_owner=child_org AND role_series=leadership AND role_rank=deputy`

## 6. 规则复用方式（Rule Bundle）

### 6.1 规则包结构

一套规则包 `bundle` 包含：

- 总分规则（模块组成、权重、权重外加减分）
- 模块规则（direct/vote/custom/extra 及明细）
- 等第划分规则（分数段、比例或混合）
- 排名规则（同分处理优先级）
- 元数据（版本号、生效状态、说明）

### 6.2 绑定关系

通过 `rule_binding` 绑定：

- `period_scope`：单周期或周期组
- `owner_scope`：全局/组织类型/具体组织
- `segment_code`：对象分群
- `bundle_id`：规则包
- `priority`：优先级（越具体优先级越高）

### 6.3 继承与覆盖

支持三级复用：

- 基础规则包：集团通用模板
- 派生规则包：继承基础包，仅调整局部参数
- 绑定覆盖：仅针对某组织/周期/分群覆写

优先级建议：

1. 指定组织 + 指定周期 + 指定分群
2. 指定组织类型 + 周期组 + 指定分群
3. 全局 + 周期组 + 指定分群
4. 全局默认

## 7. 适配当前业务场景

以现有描述为例：

- 集团考核：
  - `self + department`（集团部门及成员）
  - `child_org + leadership_team`（权属企业领导班子及成员）
- 权属企业考核：
  - `self + department`（本企业部门及成员）
  - `child_org + leadership_team`（次级企业领导班子及成员）

可见两者复用同一组 Segment，仅 owner 不同，不需要再拆两套冗余分类。

## 8. 数据模型建议（新增）

建议新增（或重构）以下核心表：

- `assessment_object_profiles`：对象画像维度
- `assessment_segments`：分群定义
- `rule_bundles`：完整规则包
- `rule_bundle_modules`：规则包模块
- `rule_bundle_grades`：规则包等第
- `rule_bindings`：owner+period+segment 到 bundle 的绑定
- `rule_bundle_inheritance`：继承关系

说明：

- `assessment_rules` 可逐步转为 `rule_bundles + rule_bindings`。
- 原 `object_category` 可短期保留作为兼容字段，由画像映射得出。

## 9. 渐进迁移方案

### 阶段一：兼容引入

- 保留现有 `assessment_objects.object_category`
- 新增对象画像字段或画像表
- 新增 `segment_code` 计算逻辑（读时计算或写入冗余）

### 阶段二：规则双轨

- 新增 `rule_bundles` 与 `rule_bindings`
- 先在“新建规则”走新模型
- 老规则继续按旧模型读取

### 阶段三：统一计算入口

- 计算引擎改为优先按 `owner + period + segment` 选规则
- 若无命中，回退旧规则维度

### 阶段四：完成迁移

- 批量迁移历史规则为 `bundle + binding`
- 前端规则页面仅保留分群与规则包配置
- 逐步下线旧 `object_category` 驱动逻辑

## 10. 风险与控制

- 风险一：分群表达式过度复杂  
  控制：限制为结构化条件，不开放无限表达式。

- 风险二：继承链过深导致排查困难  
  控制：限制继承层级（建议 <= 2），并提供“最终生效规则展开”视图。

- 风险三：新旧规则并行阶段行为不一致  
  控制：同一对象同一周期运行双算对比，结果差异告警。

## 11. 落地建议

- 先实现 `Segment + Bundle + Binding` 最小闭环。
- 第一批仅覆盖最常用的 6~8 个分群，避免一次性过度抽象。
- 规则管理界面按“对象分群”配置，隐藏底层维度细节。
- 导出审计中增加“命中的分群/绑定/规则包版本”字段，保证可追溯。

---

本方案目标是在不牺牲可维护性的前提下，支持集团多层级、跨主体、跨周期的规则复用与差异化配置。随着组织层级扩展，规则模型仍可稳定演进。
