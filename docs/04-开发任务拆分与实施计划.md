# AssessV2 开发任务拆分与实施计划（重构基线）

- 版本：v3.0
- 日期：2026-03-18
- 基线：当前“场次化 + 规则拷贝绑定”实现

## 1. 目标

在已完成核心重构的基础上，继续收口遗留代码与测试，形成可长期维护的稳定版本。

## 2. 已完成任务（Done）

## 2.1 数据模型与迁移

- 完成业务迁移 `0002_assessment_session_rebuild`。
- 建立场次主模型及规则拷贝绑定模型。
- 清理旧年度主表链路（数据库层）。

## 2.2 后端接口

- 场次管理接口全量落地。
- 规则文件库与绑定接口落地。
- Root/Admin 权限边界落地。

## 2.3 前端页面

- “周期管理”升级为“考核管理”。
- “总分管理 + 模块管理”合并为“规则管理”。
- TopBar 统一为“场次/周期/对象类型”。
- 结构化规则编辑器落地（两级分数模块 + 额外加减分不可删）。

## 3. 待完成任务（Next）

1. 清理历史未使用服务与模型文件。
2. 清理或标记未启用路由与接口（如历史分类接口）。
3. 增补场次主流程的集成测试：
   - 创建场次
   - 修改周期与分组
   - 对象增删与重置
   - 规则绑定拷贝

## 3.2 P1（建议 2-3 周）

1. 补全前端 E2E：
   - TopBar 上下文切换一致性
   - 规则绑定上下文联动
   - Admin 与 Root 的操作差异
2. 审计日志补充字段规范（统一事件码与对象标识）。
3. 规则 JSON Schema 抽象与后端校验前置。

## 3.3 P2（建议 3-4 周）

1. 增加规则版本回溯与比较。
2. 增加对象快照与组织变更影响分析。
3. 完善部署与运维文档（备份恢复演练流程）。

## 4. 任务拆分

## 4.1 后端

- `BE-01`：服务层历史代码清理
- `BE-02`：路由收敛与接口可用性标注
- `BE-03`：规则 JSON 校验器
- `BE-04`：审计事件标准化
- `BE-05`：集成测试补齐

## 4.2 前端

- `FE-01`：上下文容错与提示增强
- `FE-02`：规则编辑 UX 提升
- `FE-03`：权限驱动按钮行为统一
- `FE-04`：E2E 测试编排

## 4.3 测试与交付

- `QA-01`：接口契约回归
- `QA-02`：权限矩阵回归
- `QA-03`：备份恢复演练
- `QA-04`：文档一致性审查

## 5. 里程碑建议

- M1：主流程测试闭环（2 周）
- M2：遗留代码清理（1-2 周）
- M3：质量与可运维增强（2 周）

## 6. 验收标准（Definition of Done）

每项任务完成需满足：

1. 代码实现 + 自动化测试。
2. 权限与组织范围校验覆盖。
3. 审计日志可追踪关键操作。
4. 文档同步更新（接口/表结构/交互）。
5. 不破坏当前场次主流程。

## 7. 风险与应对

- 风险：历史代码与新模型混用导致误调用。
  - 应对：在服务层与路由层双重收敛。
- 风险：权限边界在前端仅靠按钮隐藏。
  - 应对：以后端校验为准，前端只做体验层提示。
- 风险：规则 JSON 结构被手工编辑破坏。
  - 应对：引入 Schema 校验与保存前验证。

---

本计划以“稳定当前重构成果、清理遗留、补齐测试”为主，不再回退到旧年度模型路线。
