## 1. 目标与范围

本计划用于把规格说明（01）和数据库设计（02）转成可执行开发任务，目标是：
1. 先消除文档和数据模型冲突，避免后续返工。
2. 按“可运行增量”分阶段交付，不做一次性大爆发开发。
3. 每个阶段都可验收（接口、页面、测试、数据一致性）。

当前代码基线为“框架骨架”：
- 后端已有 Gin 路由占位、JWT 骨架、SQLite 连接。
- 前端已有登录页、主布局、模块占位页面。
- 业务模型、迁移、核心 API、规则引擎尚未落地。

## 2. 关键一致性修正（P0，必须先做）

以下问题若不先修正，会直接阻断开发或导致设计返工。

### 2.1 规格与数据库设计冲突清单

1. **分数模块类型不一致**
- 01要求：直接录入、投票、自定义规则计算（表达式）。
- 02当前：`score_modules.module_code` 只有 `direct/vote/extra`。
- 处理：新增“自定义规则模块”能力（建议 `custom`），并补充表达式字段与校验策略。

2. **模块建模粒度不足**
- 01要求：单规则最多10个模块，支持同类模块多个实例。
- 02当前：`UNIQUE(rule_id, module_code)`，导致每种类型最多一个模块。
- 处理：改为 `UNIQUE(rule_id, module_key)` 或 `UNIQUE(rule_id, sort_order)` + 业务唯一约束。

3. **加减分归属冲突**
- 01要求：权重外加减分“不属于任何模块”。
- 02当前：`extra_points` 依赖 `module_id`，并把 `extra` 作为模块代码。
- 处理：保留 `extra` 业务概念，但明确其“不占用权重、不参与权重折算”；`extra_points` 独立存储。

4. **计算结果结构不足**
- 01要求：存储“各模块分数 + 总分 + 排名依据”，支持复杂依赖。
- 02当前：`calculated_scores` 固定 `direct_score/vote_score/extra_points` 三列，不支持多模块扩展。
- 处理：改为通用结构（JSON明细 + 聚合字段）或新增 `calculated_module_scores` 子表。

5. **投票模型与业务语义不一致**
- 业务确认：投票不是直接打分，而是投“优/良/中/差”档位。
- 02当前：`vote_records` 保存数值分数。
- 处理：改为保存档位，并按固定公式计算投票模块分：
  - `Σ_g ( Σ_o ( score(o) × w(g) × count(g,o) ) / total(g) )`

6. **初始化脚本与v2.1设计冲突**
- 02中默认管理员插入仍使用 `users.role_id`，但v2.1已改为 `user_roles` 多对多。
- 处理：修正初始化脚本为 `users + user_roles` 两步插入。

7. **备份保留策略不一致**
- 01要求：自动备份保留最近7天。
- 02初始化值：`backup.retention_days=30`。
- 处理：默认7天，可在系统设置中配置。

8. **认证默认账号口径不一致**
- 决策：统一为 `root/#2026@hdwl`，最高权限角色为 `root`。
- 处理：初始化脚本、前后端登录骨架、文档提示统一。

### 2.2 本阶段交付物

1. `docs/02-数据库设计文档.md` 修订至 v2.2（与01一致）。
2. `docs/数据库设计更新说明.md` 增补 v2.2 变更。
3. 迁移方案文档：初始建库 SQL + 版本升级 SQL（避免仅靠 AutoMigrate）。
4. 架构决策记录（ADR）：规则引擎建模、计算结果建模、权限范围建模。

## 3. 开发里程碑（建议顺序）

## M0（1-2天）文档冻结与工程底座

1. 完成 P0 一致性修正。
2. 建立迁移目录（`backend/migrations`）与迁移执行器。
3. 建立统一错误码、日志上下文、请求追踪ID。
4. 建立测试底座：
- 后端：单元测试 + API集成测试。
- 前端：组件测试与基础E2E骨架。

**验收**：可一键初始化数据库并跑通健康检查、登录、基础测试流水线。

## M1（3-5天）认证与权限（可真实登录）

1. 实现 `users/roles/user_roles/user_organizations` 数据访问层。
2. 登录改为数据库认证（bcrypt）；签发 JWT，包含用户ID、角色、组织范围摘要。
3. 实现 RBAC 鉴权中间件与组织范围过滤中间件。
4. 实现基础用户管理 API：查询用户、重置密码、启停用户。

**验收**：
- 不同角色访问同一接口返回正确（允许/拒绝）。
- 首次登录强制改密生效。
- 审计日志记录登录/登出/改密。

## M2（4-6天）组织架构与年度周期

1. 落地 `organizations/departments/position_levels/employees/employee_history`。
2. 实现组织树查询、部门与人员 CRUD、人员调动并写历史。
3. 落地 `assessment_years/assessment_periods/assessment_objects`。
4. 年度创建支持“复制上年基础数据”（先做组织与规则骨架复制）。

**验收**：
- 创建年度自动生成5个周期。
- 周期状态流转可控，锁定后禁止变更。
- 停用组织不再出现在可考核对象生成范围。

## M3（4-6天）规则配置与模板

1. 落地 `assessment_rules/score_modules/vote_groups/rule_templates`。
2. 支持规则维度配置（年度+周期+对象类型+对象分类）。
3. 实时校验模块权重和=1.0000。
4. 支持 Q1-Q4 复用规则，YEAR_END 单独配置。
5. 自定义表达式模块：表达式保存、语法校验、变量白名单校验。

**验收**：
- 能创建并保存一套完整规则（含投票组与表达式模块）。
- 模板可保存、复用、跨年度复制。

## M4（4-6天）分数采集（直接录入+投票+加减分）

1. 直接录入：单条/批量，范围校验、重复校验、录入人记录。
2. 投票：任务生成、草稿保存、提交锁定、管理员重置。
3. 加减分：范围校验（-20~+20）、原因必填、可选审批。

**验收**：
- 完整走通“发起投票→投票→统计”链路。
- 录入与投票均有审计日志。

## M5（4-6天）计算引擎与排名

1. 实现计算触发（数据变更自动 + 手工重算）。
2. 实现依赖图（DAG）与拓扑计算顺序。
3. 输出模块级得分、总分、排名、计算明细。
4. 实现同分排序策略与分组排名。

**验收**：
- 200对象规模下计算响应 <3秒（目标指标）。
- 个人依赖团体、年终依赖季度等场景结果正确。

## M6（4-6天）导入导出与报表

1. Excel 模板下载、校验预览、确认导入、结果统计。
2. 导出：汇总表、明细表、投票统计、组织架构（多Sheet）。
3. 报表中心：排名、对比、趋势、分布图（ECharts）。

**验收**：
- 1000条导入 <10秒。
- 导出文件结构、样式、数据准确性通过验收样例。

## M7（3-5天）备份恢复、审计、系统设置

1. 自动备份（每日2:00）+ 手动备份 + 恢复（二次确认）。
2. 审计日志查询、过滤、详情对比、单条回滚能力。
3. 系统设置（基础/考核/安全）与配置生效链路。

**验收**：
- 自动备份计划可稳定执行。
- 关键操作可追溯、可审计。

## M8（2-4天）发布准备（Tauri）

1. 联调前后端与 Tauri 容器。
2. 打包脚本、版本号策略、增量更新配置。
3. 安装包冒烟测试（Windows/macOS）。

**验收**：
- 可安装、可启动、可升级、可备份恢复。

3. `BE-03` 认证鉴权：登录、改密、重置、JWT、RBAC、数据范围。
4. `BE-04` 组织与人员：组织树、人员调动历史、批量导入服务。
5. `BE-05` 年度与周期：创建、复制、状态流转与锁定约束。

7. `BE-07` 分数采集：录入、投票任务、投票记录、加减分审批。

10. `BE-10` 备份审计：备份计划、恢复、审计查询、回滚。

4. `FE-04` 年度周期页面：年度创建、复制、状态切换。
5. `FE-05` 规则配置页：模块拖拽排序、权重和校验、表达式编辑器。
6. `FE-06` 分数录入与投票页：任务列表、草稿、提交、明细。
7. `FE-07` 计算中心：重算入口、进度与结果明细。
8. `FE-08` 报表中心：筛选器、图表、导出。
9. `FE-09` 备份审计与系统设置页面。

### 4.3 QA任务包

1. `QA-01` 需求用例矩阵（01每条需求对应测试点）。
2. `QA-02` 接口自动化（鉴权、边界、幂等、并发）。
3. `QA-03` 计算准确性用例（权重、加减分、同分排名、依赖计算）。
4. `QA-04` 导入导出回归用例。

## 7. 风险与应对

1. **规则引擎复杂度高**
- 风险：表达式、依赖计算、性能目标冲突。
- 应对：先做可观测版本（详细计算明细+审计），再优化性能。

2. **权限模型复杂（多角色+多组织）**
- 风险：越权或漏权。
- 应对：权限判断集中化，禁止在业务层散落判断逻辑。

3. **SQLite并发与大批量导入**
- 风险：锁竞争与导入性能波动。
- 应对：导入分批事务 + 索引策略 + WAL优化。

4. **文档漂移**
- 风险：实现和文档再次分叉。
- 应对：每个里程碑结束同步更新文档与初始化脚本。

## 8. Definition of Done（统一完成标准）

每项任务完成必须同时满足：
1. 代码实现 + 单元测试 + 接口测试。
2. 关键流程有审计日志。
3. 错误场景有可读提示。
4. 文档同步（接口文档/数据字典/变更说明）。
5. 通过评审并可在本地一键运行验证。
