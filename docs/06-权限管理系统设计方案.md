# AssessV2 权限管理系统设计方案（当前实现）

- 版本：v3.0
- 日期：2026-03-18
- 适用：后端鉴权中间件 + 服务层范围控制

## 1. 设计原则

- 功能权限与数据范围分离。
- Root 具备全局能力，Admin 受组织范围限制。
- 前端按钮可见性不等于最终权限，后端为唯一裁决。
- 所有关键写操作必须记录审计日志。

## 2. 认证与身份模型

## 2.1 JWT Claims

令牌中包含：

- `uid`（用户ID）
- `username`
- `roles`
- `permissions`
- `orgScopes`
- `permissionBindings`

说明：

- `roles` 用于角色归一化权限判断。
- `permissions` 支持补充粒度控制。
- `orgScopes` 用于组织范围判定。

## 2.2 角色标准化

标准角色：

- `root`
- `assessment_admin`
- `leader`
- `staff`

历史角色映射：

- `viewer -> staff`
- `group_admin/company_admin/dept_admin -> assessment_admin`

## 3. 权限判断链路

请求进入后端时按顺序执行：

1. `RequireJWT`：认证
2. `RequirePermission`：功能权限点
3. 服务层组织范围校验：数据可访问边界

组织范围核心函数：

- `resolveAdminOrganizationID`
- `ensureAssessmentOrganizationScope`

逻辑要点：

- Root 直接放行。
- Admin 必须命中主组织（或可回退组织）范围。

## 4. 权限点矩阵（核心）

## 4.1 场次与规则

- `assessment:view`：查看场次与场次详情
- `assessment:update`：创建/修改场次、周期、对象分组、对象
- `rule:view`：查看规则文件与绑定
- `rule:update`：创建/编辑/删除规则、执行绑定

## 4.2 组织与系统

- `org:view` / `org:update`
- `backup:view` / `backup:update`
- `audit:view` / `audit:rollback`
- `setting:view` / `setting:update`

Root-only：

- 用户与角色组管理
- 系统设置页面访问

## 5. 业务边界矩阵（Root vs Admin）

## 5.1 场次

- Root：
  - 可创建任意组织场次
  - 可查看/修改任意组织场次
- Admin：
  - 仅可创建自己组织场次
  - 仅可查看/修改自己组织场次

## 5.2 规则文件

基础规则（`isCopy=false`）：

- Root：可创建、查看、删除
- Admin：可查看、可隐藏/恢复，不可创建/删除

拷贝规则（`isCopy=true`）：

- Root：可查看、编辑、删除
- Admin：仅可操作 `ownerOrgId=自己组织` 的拷贝规则

## 5.3 规则绑定

- Root：可在任意组织场次上下文绑定
- Admin：仅可在自己组织场次上下文绑定

绑定动作固定行为：

- 总是创建拷贝规则
- 绑定始终指向拷贝规则

## 6. 前后端协作约定

前端通过以下字段控制操作按钮：

- `canEdit`
- `canDelete`
- `hiddenByCurrentOrg`

说明：

- 这些字段由后端根据当前用户权限与组织范围计算。
- 前端仅用于体验控制，后端仍会二次校验。

## 7. 审计要求

以下操作应入审计：

- 创建/修改场次
- 替换周期/对象分组/对象
- 规则创建/更新/删除
- 规则绑定（含源规则ID与拷贝规则ID）
- 用户与角色组管理操作

## 8. 异常语义与建议

常见权限失败返回：

- `401`：未认证或认证上下文缺失
- `403`：权限不足或组织范围不匹配
- `404`：目标资源不存在

建议前端提示：

- 明确区分“无权限”与“资源不存在”
- 对范围失败提示“仅可操作本组织数据”

## 9. 当前风险与建议

- 风险：历史接口残留可能绕开新主流程。
  - 建议：对历史接口统一加“废弃”标记或删除。
- 风险：组织范围定义依赖 `orgScopes` 数据质量。
  - 建议：增加用户组织绑定一致性校验任务。
