# AssessV2 组织级逻辑备份与恢复设计方案

- 版本：v1.0
- 日期：2026-03-22
- 状态：设计中（待开发）

## 1. 设计背景

当前备份能力是整库快照（SQLite 全量备份）。该方式适合灾难恢复，但不适合“按组织最小化数据备份”：

- 受限管理员下载备份时可能接触全量数据。
- 无法按组织做迁移、分发、回滚。
- 权限模型无法表达“仅某组织范围”的备份操作。

因此新增“组织级逻辑备份包”，与现有全量备份并存。

## 2. 方案目标

1. 支持按组织（含子组织闭包）导出业务数据包。
2. 支持按组织范围恢复，避免整库覆盖。
3. 保留现有全量备份能力，不替代灾难恢复。
4. 备份/恢复全流程可校验、可审计、可回滚。

## 3. 备份类型

- `full_snapshot`：现有全量快照（保留，灾备用途）。
- `org_logical`：新增组织级逻辑备份包（本方案）。

## 4. 组织范围定义

输入参数：`rootOrganizationId`

默认策略：递归闭包

- `scopedOrganizationIds = rootOrganizationId + 所有子组织`

说明：

- 闭包可避免主从组织数据断裂。
- 非 Root 用户发起备份时，`rootOrganizationId` 必须在其授权组织范围内。

## 5. 数据边界

## 5.1 V1 纳入数据

组织域：

- `organizations`（id 在范围内）
- `departments`（organization_id 在范围内）
- `employees`（organization_id 在范围内）
- `employee_history`（employee_id 在范围内）

考核域（当前主流程）：

- `assessment_sessions`（organization_id 在范围内）
- `assessment_session_periods`
- `assessment_object_groups`
- `assessment_session_objects`
- `assessment_object_module_scores`
- `rule_files`
- `assessment_rule_bindings`（若表存在且业务在用）

## 5.2 V1 不纳入数据

- `system_settings`、`position_levels`
- `audit_logs`、`backups`
- accounts 库（`users/roles/user_roles/user_organizations/...`）

## 5.3 特殊处理

`employee_history` 可能引用范围外组织/部门外键。

V1 规则：

- 保留历史记录本身；
- 将范围外 `old/new organization_id` 与 `old/new department_id` 置空；
- 在 manifest 记录 `sanitizedHistoryRefsCount`。

## 6. 权限模型

建议新增权限点：

- `backup:org:view`
- `backup:org:update`

角色策略：

- Root：可操作任意组织包。
- assessment_admin：仅可操作其授权范围内组织包。

额外约束：

- 组织包下载与恢复均做“记录级组织范围校验”。
- 不允许通过猜测 `backupId` 越权下载或恢复。

## 7. API 设计

## 7.1 创建组织包

- `POST /api/backup/org-packages`
- 请求体：
  - `rootOrganizationId: number`
  - `description: string`
  - `includeEmployeeHistory: boolean`（默认 `true`）

## 7.2 查询组织包列表

- `GET /api/backup/org-packages?page=1&pageSize=20&rootOrganizationId=...`

## 7.3 下载组织包

- `GET /api/backup/org-packages/:id/download`

## 7.4 恢复组织包

- `POST /api/backup/org-packages/:id/restore`
- 请求体：
  - `confirmText: "CONFIRM_ORG_RESTORE"`
  - `mode: "replace_scope"`（V1 仅支持该模式）
  - `targetRootOrganizationId: number`（V1 要求与包内一致）

## 8. 存储与格式

## 8.1 备份记录扩展

扩展 `backups` 表（推荐）：

- `content_type`：`full_snapshot | org_logical`
- `scope_type`：`global | organization`
- `scope_org_id`：组织包根组织 ID（可空）
- `format_version`：如 `orgpkg-v1`
- `checksum_sha256`
- `manifest_json`

## 8.2 文件格式

推荐 `tar.gz`，包含：

- `manifest.json`
- `data.db`（子集 SQLite）

`manifest.json` 建议字段：

- `packageType`
- `formatVersion`
- `createdAt`
- `rootOrganizationId`
- `scopedOrganizationIds`
- `tableRowCounts`
- `businessSchemaVersion`
- `checksum`

## 9. 导出流程

1. 鉴权 + 组织范围校验。
2. 计算 `scopedOrganizationIds`。
3. 创建临时 `data.db` 并初始化子集表结构。
4. 按依赖顺序导出数据到 `data.db`。
5. 生成 `manifest.json`。
6. 打包压缩，计算 SHA-256。
7. 写 `backups` 记录，落审计日志。

导出过滤核心：

- `organizations.id IN scopedOrganizationIds`
- 其余表按 `organization_id` / `assessment_id` / `object_id` 逐级关联过滤。

## 10. 恢复流程

## 10.1 恢复前校验

1. 权限与组织范围校验。
2. 包完整性校验（文件存在、checksum、manifest）。
3. 目标组织校验（V1 要求与包内 root 组织一致）。
4. 冲突预检查（唯一键、引用完整性）。

## 10.2 安全护栏

- 恢复前自动创建一次 `full_snapshot`（`before_restore`）。
- 恢复必须输入确认文本。
- 全流程记录审计日志。

## 10.3 V1 恢复模式：`replace_scope`

事务内执行（必要时短时关闭 FK，提交前 `foreign_key_check`）：

1. 计算目标组织闭包。
2. 按依赖逆序删除目标范围旧数据。
3. 按依赖正序导入包内数据。
4. 完整性检查通过后提交。

删除顺序建议（逆序）：

1. `assessment_object_module_scores`
2. `assessment_session_objects`
3. `assessment_rule_bindings`
4. `rule_files`
5. `assessment_object_groups`
6. `assessment_session_periods`
7. `assessment_sessions`
8. `employee_history`
9. `employees`
10. `departments`
11. `organizations`（根组织建议 upsert，不直接硬删）

## 11. 回滚与可观测性

回滚策略：

- 恢复失败自动事务回滚；
- 恢复后逻辑异常可回退到 `before_restore` 全量快照。

监控指标建议：

- 导出/恢复耗时
- 包大小
- 各表行数
- 校验失败次数
- FK 检查失败次数

日志脱敏：

- 不记录密码、token、原始敏感 JSON；
- 文件路径仅记录相对名或备份 ID。

## 12. 分阶段实施

## 12.1 Phase 1（最小可用）

- 增加 `org_logical` 类型与记录扩展字段。
- 完成“创建/列表/下载组织包”接口。
- 完成恢复 `replace_scope`（同组织 ID）。
- 前端新增“组织备份包”页（Root 可见）。

## 12.2 Phase 2（增强）

- assessment_admin 范围化开放。
- 增加 `dry-run` 恢复预检查接口。
- 增加差异摘要输出（新增/更新/删除行数）。

## 12.3 Phase 3（跨环境迁移）

- 支持目标组织映射（ID remap）。
- 支持冲突策略：`skip` / `update` / `fail`。
- 可选增加包签名验签。

## 13. 验收标准

1. Root 导出组织 A 包，不含组织 B 数据。
2. Admin 越权导出/下载/恢复返回 `403`。
3. 恢复前自动生成 `before_restore` 全量备份。
4. 恢复失败不污染现有数据。
5. 恢复后 `foreign_key_check` 通过。
6. 审计记录完整且不泄露敏感信息。

## 14. 决策建议

建议采用双轨备份：

- `full_snapshot`：灾难恢复专用；
- `org_logical`：组织级导出与最小范围恢复。

该策略兼顾安全边界与运维可用性，且能在现有架构中分阶段落地。
