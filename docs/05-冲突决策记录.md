# AssessV2 冲突决策记录（重构版）

- 版本：v3.0
- 更新日期：2026-03-18
- 状态：生效

## 1. 决策清单

## D-001：业务主实体从“年度”切换为“考核场次”

- 决策：采用 `assessment_sessions` 作为主实体。
- 原因：年度模型难以承载组织差异化与多场次并行。
- 结果：周期、对象分组、对象列表均挂载到场次。

## D-002：Root/Admin 与考核对象解绑

- 决策：Root/Admin 作为管理角色，仅受组织范围约束，不再绑定考核对象实体。
- 原因：对象与权限耦合导致维护复杂且边界混乱。
- 结果：权限模型改为“角色 + 组织范围”。

## D-003：取消“同年同组织唯一场次”约束

- 决策：不增加 `(year, organization_id)` 唯一限制。
- 原因：用户已明确要求忽略该限制。
- 结果：同组织同年允许多个场次。

## D-004：周期与对象分组改为场次内配置

- 决策：不再使用全局周期模板/对象类型模板驱动运行。
- 原因：组织业务差异大，通用模板会造成大量例外处理。
- 结果：每场次独立维护周期与对象分组。

## D-005：规则绑定采用“拷贝后绑定”

- 决策：每次绑定都创建规则拷贝，绑定指向拷贝。
- 原因：避免跨组织共享规则被相互改写。
- 结果：组织可独立演进规则，互不影响。

## D-006：规则编辑权限限定到拷贝文件

- 决策：基础规则不可被 Admin 直接编辑；Admin 仅编辑本组织拷贝。
- 原因：控制影响范围，保证全局规则基线稳定。
- 结果：Root 可管理所有规则；Admin 管理自己组织拷贝。

## D-007：规则管理页面结构化编辑

- 决策：总分模块与模块规则合并到同一页面，结构化编辑规则 JSON。
- 原因：减少页面分裂与上下文切换成本。
- 结果：形成“规则文件库 + 结构化编辑 + 绑定”一体化页面。

## D-008：额外加减分模块固定保留

- 决策：规则内固定存在 `isExtra=true` 模块，不可删除。
- 原因：与总分公式耦合，删除会破坏统一计算口径。
- 结果：UI 与服务层均按固定模块设计。

## D-009：TopBar 上下文统一为三维

- 决策：全局选择器固定为“场次/周期/对象类型”。
- 原因：规则绑定与评分场景均依赖这三维。
- 结果：页面通过同一 context store 共享状态。

## D-010：旧年度主表直接下线，不做兼容

- 决策：通过迁移删除旧核心表链路，不保留双轨运行。
- 原因：用户要求“完全重建并清理旧处理逻辑”。
- 结果：数据库主流程仅保留场次模型。

## 2. 决策影响

正向影响：

- 业务边界更清晰。
- 组织间规则污染问题被消除。
- 前端交互路径统一，学习成本降低。

代价与风险：

- 历史代码残留增加维护噪音。
- 旧文档口径全部失效，必须同步重写。
- 需要更多自动化测试保障新模型稳定。

## 3. 后续执行要求

- 所有新需求先判断是否符合“场次中心”原则。
- 涉及规则变更必须经过“拷贝规则”路径，不得绕过。
- 任何新接口上线前必须补齐权限与组织范围校验。
